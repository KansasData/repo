Quarto 1.9.x Bug: {{\< meta \>}} Shortcodes in {=html} Blocks Return Only First Word

Summary

In Quarto 1.9.27 (and likely all 1.9.x versions), {{\< meta key >}} shortcodes inside {=html} raw HTML blocks silently truncate multi-word string metadata values to the first word only. This is a regression from Quarto 1.8.x, where this worked correctly.

Symptoms

Given a .qmd file with:

---
project-title: "Surface Water Drinking Water"
datatype: "Tabular Data"
---

And inside a {=html} block:

<h2>{{\< meta project-title >}}</h2>
<div>{{\< meta datatype >}}</div>

The rendered HTML shows:

<h2>Surface</h2>       <!-- only first word -->
<div>Tabular</div>     <!-- only first word -->

Single-word values (e.g., partner: "ARISE", year: "2026") are unaffected. List-format metadata values (e.g., datacategory: - "...") are also unaffected.

Root Cause

In Quarto 1.9.x, the Lua shortcodes filter (main.lua) processes RawBlock nodes (which includes {=html} blocks) using code_handler, which invokes shortcodes in “text” context. The handleMeta function in text context contains this logic:

if type(optionValue) == "table" and #optionValue > 0 and optionValue[1].t == "Str" then
    return optionValue[1].text   -- returns ONLY the first word!

A YAML plain string like "Surface Water Drinking Water" is parsed by Pandoc as MetaInlines — a Lua table whose first element is Str "Surface". This condition fires and returns only "Surface".

This behavior was NOT present in Quarto 1.8.x, where {=html} raw blocks did not go through this “text” context code path.

Who Is Affected

Any Quarto project that: 1. Uses {{\< meta custom-key >}} shortcodes inside {=html} raw blocks, AND 2. The metadata value is a multi-word plain string (not a list)

Pages rendered with Quarto 1.8.x are unaffected until re-rendered with 1.9.x. Re-rendering with 1.9.x will silently break them.

Workaround

Change multi-word plain string metadata values to single-element YAML lists. List items are stored as MetaList (not MetaInlines), so the bug condition does not fire.

Before (broken in 1.9.x):

project-title: "Surface Water Drinking Water"
datatype: "Tabular Data"
dataset-title: "Surface Water Drinking Water"

After (works in 1.9.x):

project-title:
  - "Surface Water Drinking Water"
datatype:
  - "Tabular Data"
dataset-title:
  - "Surface Water Drinking Water"

The shortcode {{\< meta project-title >}} in the {=html} block then correctly renders the full string. This is also consistent with how datacategory and keyword are already stored as lists in this project.

This Project’s Status

As of 2026-03-23: Quarto 1.9.27 was uninstalled; RStudio now uses the bundled 1.8.25. All list-format workarounds have been reverted to plain strings. New pages should be written with plain strings from the start — no workaround needed on 1.8.25, and they will render correctly once the bug is fixed upstream in a future 1.9.x release.

  • project/projects/project_SurfaceWaterDrinkingWater.qmd — reverted (project-title, datatype back to plain strings)
  • project/datasets/data_SurfaceWaterDrinkingWater.qmd — reverted (dataset-title back to plain string)
  • project/projects/project_DisabilityAndHealthData.qmd — reverted (project-title, partner, datatype back to plain strings; note: partner was also a list workaround since “Industry Advisory Board” is multi-word)
  • No data_DisabilityAndHealthData.qmd exists (no dataset page for this project)

Quarto Version Context (this machine)

RStudio is currently using 1.9.27 (global install takes precedence)

RStudio’s bundled Quarto is not special — it is simply the version current at the time RStudio was packaged. In newer RStudio versions, if a newer Quarto is found on the system PATH, it is preferred over the bundled version.

To check/change which Quarto RStudio uses: Tools → Global Options → Quarto

Reporting

This appears to be an unintentional regression. It should be reported at: https://github.com/quarto-dev/quarto-cli/issues

A minimal reproduction: a .qmd file with a multi-word string metadata field (e.g., my-title: "Hello World") referenced via {{\< meta my-title >}} inside a {=html} block, rendered with Quarto 1.9.x.
